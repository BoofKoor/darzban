# Marzban Fork — v0.9.0 Decisions

**Purpose:** answers to the 12 open questions raised in `CODEBASE_MAP.md` §7. This document is the source of truth for v0.9.0 scoping. All future Claude Code prompts should reference this file.

**Status:** approved, locked for v0.9.0. Future versions (v0.10+) may revisit out-of-scope items.

---

## Guiding principles

1. **Compatibility is non-negotiable.** Users on Marzban 0.8.4 must be able to upgrade to our v0.9.0 with no database migration loss, no subscription URL changes, and no config-file changes. Additive only.
2. **Stability over features.** v0.9.0 is a maintenance release. No new user-visible features unless they fix a known pain point.
3. **Test what you change.** Every non-trivial change must ship with a test that would have caught the bug.
4. **No silent behavior drift.** If behavior must change, it must be documented in `CHANGELOG.md` and `MIGRATION_GUIDE.md`.

---

## v0.9.0 in scope

- Test infrastructure from zero (pytest, fixtures, CI)
- Pinned Xray binary with `XRAY_VERSION` override
- Node reconnect logic: exponential backoff + circuit breaker
- Logging improvements (ERROR level + exception text for node failures)
- `@app.on_event` → `lifespan` migration
- Decouple `app.db.models → app.xray` import (prerequisite for unit-testable models)
- Fix `usage_coefficient` not persisted on node create
- Fix mutable-default-arg bug in `review_users.py:20`
- Tighten bare `except:` sites that mask errors
- Per-client transport capability matrix in subscription generator
- `pcs/vcn` parameters added for Xray `allowInsecure` deprecation (UTC 2026.06.01)
- Single-worker enforcement (explicit error if `workers > 1`)
- rpyc kept with DEPRECATED warning (drop planned for v1.0)
- Optional JSON log formatter via `LOG_FORMAT=json` env var
- Minimal frontend changes only (surface node error messages, host-incompat warnings)

## v0.9.0 out of scope (deferred to v1.0 or later)

- Async SQLAlchemy migration / async-def endpoints
- Multi-worker support, externalized scheduler, Redis state
- Frontend major upgrades (React 19, Vite 5, TS 5)
- New protocols beyond what Xray v26.x ships
- UI redesign
- 2FA, audit log, webhooks, device limit, RBAC (these were bucket-3 differentiators — they come in v0.10+)
- Dropping rpyc support
- New features added by `xray-core` that require backend orchestration changes (XHTTP is already in Xray; we just need the subscription generator to handle it)

## Decisions on the 12 open questions

### Q1 — Test suite in scope?
**Yes, top priority.** Without tests, a "stability release" is a contradiction.

- **Stack:** `pytest` + `pytest-asyncio` + `pytest-mock` + `httpx.AsyncClient` for API tests + SQLite in-memory fixture for DB
- **Priority order:** subscription format correctness → user state transitions (`review()`) → node reconnect logic
- **Coverage targets for v0.9.0:** 70% on `app/subscription/`, 80% on `app/jobs/review_users.py`, 60% on `app/xray/node.py` + `app/xray/operations.py`
- **Golden-file approach** for subscription tests: one fixture per `(format, scenario)` pair; assert byte-equality after deterministic ordering.
- **CI:** GitHub Actions, runs on every PR, blocks merge on failure.

### Q2 — Pin Xray binary?
**Yes.**

- **Baseline version:** latest stable `v26.x` at the time of v0.9.0 work start (currently `v26.2.6`)
- **Override:** new `XRAY_VERSION` env var honored by the install script
- **Rename:** `install_latest_xray.sh` → `install_xray.sh`, accept a version argument
- **No silent auto-upgrades.** The image rebuild must produce the same Xray version unless explicitly bumped.

### Q3 — Reconnect backoff?
**Yes, exponential per-node.**

- Per-node backoff state: 1s → 2s → 4s → 8s → 16s → 32s → 64s → 128s → cap at 300s
- Reset to 1s on successful connection
- Circuit breaker: after N consecutive failures (default N=5), node enters `error` with a `retry_after` timer; health-check skips it until cooldown
- Admin-visible: real error message + retry-after surfaced via existing `Node.message` field
- Tests must cover: backoff progression, reset-on-success, circuit-breaker entry/exit

### Q4 — Per-client transport filtering?
**Yes, via a maintainer-managed capability matrix. NOT per-user configuration.**

- New module `app/subscription/client_compat.py` containing a `CLIENT_COMPAT` matrix mapping `(client_format, transport, security, header_type) → bool`
- Applied as a filter step inside `process_inbounds_and_tags` in `app/subscription/share.py`
- Behavior: incompatible (inbound × client) combinations are silently dropped from that client's subscription output (no error, just absence)
- Admin UI: on each host, show a small badge listing which clients will not receive this host (this is the only frontend touch in v0.9.0)
- Optional escape hatch: per-host boolean `force_include` to override the filter for testing (default `False`)
- This is additive — existing v2ray output unchanged because v2ray dropped nothing before either

### Q5 — Drop rpyc?
**No. Keep with deprecation.**

- Many older Marzban-node deployments still use rpyc. Dropping breaks them, which violates the compatibility promise.
- On rpyc detection: log a `DEPRECATED` warning once per node per process lifetime
- Add a `rpyc` deprecation notice in `CHANGELOG.md` and `MIGRATION_GUIDE.md` with v1.0 as the removal target
- Document the switch-to-REST procedure in `docs/migrating-from-rpyc.md`

### Q6 — `lifespan` migration?
**Yes, in v0.9.0.**

- Migrate all `@app.on_event("startup")` and `@app.on_event("shutdown")` handlers (see CODEBASE_MAP §6.7) to a single `lifespan` async context manager
- Semantics preserved: same startup order, same shutdown order
- Required for forward compatibility with Starlette ≥ 0.40

### Q7 — Decouple `db.models → app.xray`?
**Yes, as a test prerequisite.**

- The `User.inbounds` hybrid currently calls `app.xray.config.inbounds_by_protocol` directly, which forces the Xray module to be initialized before any model can be instantiated — blocks isolated unit tests of models
- Approach: introduce a thin `InboundLookup` protocol/service in `app/db/lookups.py`. Default implementation reads from `app.xray.config`. Tests can inject a fake.
- Same for the Pydantic validators in `app/models/user.py:159,180,189,257`
- **Behavior must be unchanged.** This is pure refactor.

### Q8 — Logging?
**Yes.**

- All `logger.info("Unable to connect…")` style calls in `app/xray/operations.py` move to `logger.error(...)` with `exc_info=True`
- Bare `except:` sites enumerated in §6.5 are tightened: catch only the specific expected exception type, or `except Exception` with `logger.exception(...)`
- Add optional JSON formatter via `LOG_FORMAT=json` env var (default `text` to preserve current behavior). Use stdlib only — no new dependency.

### Q9 — Async DB?
**No for v0.9.0.**

- Migration to async SQLAlchemy is a multi-week effort that touches every endpoint. Too invasive for a compatibility-first release.
- Spot-fix the two async websocket handlers (`core_logs` in `app/routers/core.py:21`, `node_logs` in `app/routers/node.py:83`) by moving their brief auth DB call to `run_in_threadpool`.
- Full async migration is a v1.0+ candidate.

### Q10 — `usage_coefficient` at create?
**Unintended bug. Fix it.**

- Add `usage_coefficient=node.usage_coefficient` to the `Node(...)` constructor call in `crud.create_node` (`app/db/crud.py:1307-1311`)
- Add a regression test

### Q11 — Multi-worker?
**Single-worker permanent for v0.9.0.**

- Add an explicit check at startup: if `workers > 1` is detected via env or CLI, log an error and exit with a clear message
- Multi-worker is a v1.0+ goal requiring: externalized scheduler (e.g. `arq` or `dramatiq`), Redis or similar for cross-worker Xray state coordination, websocket sticky-sessions

### Q12 — Frontend scope?
**Backend-stability-focused. Minimal frontend changes only.**

- Allowed in v0.9.0:
  - Surface `Node.message` content properly in the node card UI
  - Add the client-compat badge on host rows (from Q4)
- Not in v0.9.0:
  - React 18 → 19, Vite 3 → 5, TS 4.9 → 5.x
  - Dashboard redesign
  - New screens or major UX changes

## Workplan (~6 weeks)

### Week 1–2 — Foundation
- Pytest setup, conftest, SQLite in-memory fixture, GitHub Actions CI (lint + test)
- Decouple `db.models → app.xray` via `InboundLookup` protocol
- Pin Xray binary, `XRAY_VERSION` env var, rename install script
- Quick wins: fix `usage_coefficient` create bug, fix mutable default arg, tighten the highest-risk bare `except:` sites
- Add startup check forbidding `workers > 1`
- One smoke test verifying app boots, one DB model test, one subscription happy-path test → CI green

### Week 2–3 — Reliability
- Lifespan migration
- Exponential backoff + circuit breaker for node reconnect (with tests)
- Node logging → ERROR + `exc_info=True`
- Optional JSON log formatter
- rpyc deprecation warning + docs

### Week 3–4 — Subscription
- `CLIENT_COMPAT` matrix module + tests
- Filter step in `process_inbounds_and_tags`
- `pcs/vcn` params for `allowInsecure` deprecation
- Golden-file subscription tests for every (format × scenario) combination
- Minimal frontend: host-compat badges, node error message surfacing

### Week 4–5 — Hardening
- Node lifecycle tests with mocked gRPC
- Review and tests for user state transitions (`review()` paths)
- Reduce `except Exception` swallowing where it hides real failures
- `MIGRATION_GUIDE.md`, `CHANGELOG.md` finalized
- Documentation refresh

### Week 5–6 — Release
- v0.9.0-rc1 on staging
- Beta tester feedback
- Bug-fix cycle
- v0.9.0 stable: tag, Docker image push, release notes, announcement

## Definition of done for v0.9.0

- All v0.9.0 in-scope items above shipped
- CI green on `main` with ≥ the coverage targets in Q1
- A fresh `docker compose up` against the v0.9.0 image, using a v0.8.4 SQLite DB copied in, boots cleanly with no manual migration step
- A v0.8.4 user's subscription URL returns valid output in every supported client format from the v0.9.0 server
- `MIGRATION_GUIDE.md` covers every behavior change with mitigation
- No regression in the existing happy-path flows captured by Week-3 golden-file tests
